# PRD: платформа генерации интерактивных SCORM-тренажёров из скриншотов

**Версия документа:** 1.0
**Статус:** черновик для архитектурной проработки
**Адресат:** архитектор / техлид

---

## 1. Резюме

Продукт состоит из двух связанных модулей:

1. **Author Tool** — веб-редактор, в котором методист без навыков программирования загружает скриншоты реальной программы и размечает поверх них сценарий работы (клики, ввод значений, переходы между экранами).
2. **Runtime Player** — сгенерированный из сценария интерактивный тренажёр, который воспроизводит эти скриншоты как «живой» интерфейс: обучаемый кликает, вводит данные, получает подсказки и балл.

Готовый тренажёр экспортируется как **SCORM-пакет** (1.2 и 2004) для загрузки в любую LMS, а также должен корректно работать автономно (локальный запуск, тестирование до заливки в LMS).

Задача архитектора — спроектировать техническую реализацию обоих модулей и модуль экспорта, обеспечивающий совместимость со сторонними LMS.

---

## 2. Проблема и контекст

Обучение новых сотрудников работе с внутренними операционными программами сейчас требует либо доступа к тестовому контуру реальной системы (дорого, рискованно, требует поддержки IT), либо статичных инструкций/скриншотов (низкая вовлечённость, нет проверки усвоения). Нужен способ быстро — силами методиста, а не разработчиков — превращать набор скриншотов и описание сценария в интерактивную симуляцию с измеримым результатом прохождения, интегрируемую в существующую корпоративную LMS.

---

## 3. Цели (Goals)

| # | Цель | Метрика успеха |
|---|------|-----------------|
| G1 | Методист без техфона собирает рабочий тренажёр из скриншотов | Сборка тренажёра из 3–5 шагов ≤ 30 минут |
| G2 | Тренажёр корректно работает как SCORM-пакет в реальных LMS | Успешный трекинг статуса/балла минимум в 2 LMS (напр. Moodle, SCORM Cloud) |
| G3 | Тренажёр можно тестировать локально до заливки в LMS | Полный прогон на `localhost` без ошибок, с деградацией трекинга в «автономный режим» |
| G4 | Точность попадания хотспотов не зависит от разрешения экрана обучаемого | Совпадение зон клика ±2% от размеров контейнера на разных viewport |

## 4. Не цели (Non-Goals, на этом этапе)

- Запись/распознавание сценария автоматически по видео или логам реальной системы (только ручная разметка методистом).
- Генерация скриншотов синтетически (ИИ-рендер интерфейса) — только загруженные пользователем изображения.
- Полноценная авторизация/мультитенантность уровня Enterprise (роли и совместное редактирование — в бэклог v2, не MVP).
- Мобильный нативный плеер (только веб, адаптивный).

---

## 5. Пользователи и сценарии использования

**Роли:**
- **Методист (автор)** — создаёт тренажёры в Author Tool.
- **Обучаемый (сотрудник)** — проходит тренажёр в LMS или локально.
- **Администратор LMS** — загружает SCORM-пакет, смотрит отчёты о прохождении в самой LMS.

**Ключевые user stories:**

1. Как методист, я загружаю 5 скриншотов операционной программы и рисую хотспоты, чтобы описать сценарий смены юридического адреса клиента.
2. Как методист, я хочу мгновенно посмотреть предпросмотр тренажёра, не выгружая SCORM-пакет.
3. Как методист, я хочу экспортировать готовый тренажёр в ZIP и загрузить его в корпоративный Moodle.
4. Как обучаемый, я прохожу тренажёр, ошибаюсь в поле ввода, вижу подсказку, повторяю попытку и в конце вижу свой балл.
5. Как администратор LMS, я вижу в отчётах LMS, кто из сотрудников прошёл тренажёр и с каким результатом.
6. Как методист/QA, я хочу открыть собранный тренажёр на `localhost` до публикации в LMS и убедиться, что сценарий работает.

## 6. Функциональные требования

### 6.1 Author Tool

| ID | Требование |
|----|------------|
| FR-1 | Загрузка изображений (PNG/JPG/WebP), drag-and-drop, множественная загрузка, сортировка экранов |
| FR-2 | Визуальный редактор: рисование прямоугольных хотспотов поверх изображения мышью, координаты хранятся в относительных единицах (0–1 от ширины/высоты) |
| FR-3 | Типы хотспотов: `click`, `input`, `select`, `checkbox`, `wait` (минимум для MVP: `click`, `input`) |
| FR-4 | Для `input`/`select`: правило валидации ответа — точное значение / regex / любое непустое значение / список допустимых |
| FR-5 | Настройка перехода: следующий экран при успехе, поведение при ошибке (retry / hint / переход на альтернативный экран) |
| FR-6 | Назначение баллов на хотспот и порога прохождения (passing score) для всего тренажёра |
| FR-7 | Предпросмотр сценария внутри Author Tool в режиме, идентичном финальному плееру |
| FR-8 | Импорт/экспорт сценария как JSON (см. Приложение A) — для переиспользования и версионирования |
| FR-9 | Сохранение состояния проекта тренажёра (черновик, без публикации) |

### 6.2 Runtime Player

| ID | Требование |
|----|------------|
| FR-10 | Рендер экрана-скриншота с наложенными интерактивными зонами согласно координатам из сценария |
| FR-11 | Проверка действия обучаемого против правила валидации хотспота, показ подсказки/ошибки |
| FR-12 | Прогресс-бар, счётчик попыток, отображение текущего балла |
| FR-13 | Итоговый экран с результатом: балл, % с первой попытки, статус (пройдено/не пройдено) |
| FR-14 | Автономный режим: при отсутствии SCORM API — работа без ошибок, локальное хранение прогресса |
| FR-15 | Адаптивная вёрстка — хотспоты остаются точными при разных размерах окна/контейнера LMS |

### 6.3 Экспорт / SCORM

| ID | Требование |
|----|------------|
| FR-16 | Генерация `imsmanifest.xml` для SCORM 1.2 и SCORM 2004 (3rd/4th Ed.), выбор версии перед экспортом |
| FR-17 | JS-обёртка SCORM API: `findAPI`, `Initialize/LMSInitialize`, `SetValue/LMSSetValue`, `Commit/LMSCommit`, `Terminate/LMSFinish` |
| FR-18 | Трекинг: `cmi.completion_status`, `cmi.success_status`, `cmi.score.raw`, `cmi.interactions.n.*`, `cmi.suspend_data` (возобновление прохождения) |
| FR-19 | Упаковка в единый ZIP-архив корректной структуры, готовый к загрузке в LMS без ручной доработки |
| FR-20 | Автоматическая оптимизация изображений при экспорте (сжатие/ресайз) с индикацией финального веса пакета |

## 7. Нефункциональные требования

- **Производительность:** открытие тренажёра с 10–15 экранами — не более 2 сек. до интерактивности на средней конфигурации.
- **Совместимость LMS:** протестировать минимум на Moodle и SCORM Cloud (валидатор); задокументировать известные ограничения (iSpring, TalentLMS — по возможности).
- **Ограничение размера пакета:** предупреждение, если ZIP превышает 150 МБ (частый лимит корпоративных LMS).
- **Локализация:** интерфейс Author Tool и Player — RU/EN, тексты сценария — мультиязычны через отдельные JSON-локали.
- **Безопасность:** валидация MIME/размера загружаемых файлов, санитизация пользовательского текста (защита от XSS в подсказках/названиях).
- **Доступность (a11y):** текстовые инструкции к каждому шагу доступны для скринридера; управление с клавиатуры для основных действий плеера.
- **Восстановление:** обрыв соединения/закрытие вкладки не должны сбрасывать прогресс (suspend_data + локальный кэш).

## 8. Архитектура (предложение для проработки)

```
┌─────────────────────┐        ┌──────────────────────┐
│   Author Tool (SPA)  │        │  Runtime Player       │
│  React + TS          │        │  Vanilla TS bundle    │
│  - Image upload       │        │  - рендер экрана      │
│  - Hotspot editor      │───────▶│  - валидация действий │
│    (canvas/svg layer)  │  JSON  │  - SCORM API wrapper  │
│  - Scenario timeline    │ scenario│  - suspend/resume    │
└──────────┬───────────┘        └──────────┬────────────┘
           │                                │
           ▼                                ▼
   ┌───────────────┐               ┌────────────────────┐
   │ Backend / API  │               │  Export Service     │
   │ - проекты       │               │  - imsmanifest.xml   │
   │ - ассеты (S3/CDN)│              │  - image optimize     │
   │ - версии сценария │             │  - ZIP packaging       │
   └───────────────┘               └────────────────────┘
```

**Вопросы для архитектора:**
1. Author Tool — SPA с бэкендом (хранение проектов/ассетов на сервере) или полностью клиентское решение (проект хранится и переносится как `.zip`, без сервера)? Компромисс между простотой инфраструктуры и мультипользовательской работой/совместным редактированием.
2. Формат хранения ассетов — объектное хранилище (S3-совместимое) с CDN-раздачей или локальная файловая система на этапе MVP.
3. Runtime Player — единая сборка (webpack/vite → один `index.html` + `assets/`) для простоты встраивания в SCORM-пакет; стоит ли переиспользовать один и тот же бандл для всех тренажёров, подгружая `scenario.json` динамически, или собирать бандл индивидуально под каждый тренажёр.
4. Экспорт-сервис — синхронная генерация на клиенте (JS в браузере, без бэкенда) vs серверная задача (очередь, для тяжёлой оптимизации изображений).

## 9. Модель данных (черновая)

**Trainer (проект тренажёра)**
- `id`, `title`, `language`, `passing_score`, `scorm_version`, `created_at`, `updated_at`, `status` (draft/published)

**Screen (экран)**
- `id`, `trainer_id`, `image_url`, `order`, `narration`

**Hotspot**
- `id`, `screen_id`, `type`, `region {x,y,width,height}`, `expected_value`, `on_success {next_screen_id, score}`, `on_error {hint, retry, alt_screen_id}`

**Attempt/Interaction (учебная сессия — для аналитики поверх SCORM cmi.interactions, опционально во внешней БД аналитики)**
- `learner_id` (если доступен из LMS), `trainer_id`, `hotspot_id`, `result`, `attempt_number`, `timestamp`

Полная JSON-схема сценария — см. Приложение A.

## 10. Технологический стек (предложение)

| Слой | Технология | Комментарий |
|------|------------|-------------|
| Author Tool | React + TypeScript | Стейт: Zustand/Redux |
| Разметка хотспотов | SVG-слой или Canvas (fabric.js / react-konva) | Нужна точность координат в % |
| Runtime Player | Vanilla TS, без фреймворка в проде | Минимальный вес, совместимость с iframe LMS |
| Экспорт SCORM | Node-скрипт: генерация manifest + `jszip` для упаковки | Может выполняться и в браузере, и на сервере |
| Хранилище (если серверный вариант) | Postgres (проекты/сценарии) + S3-совместимое хранилище (ассеты) | — |
| Валидация SCORM-пакета | SCORM Cloud API / локальный валидатор перед релизом | Часть CI-пайплайна |

*Финальный выбор — на усмотрение архитектора, список задаёт ограничения (лёгкий рантайм, статическая упаковка, совместимость с iframe LMS).*

## 12. Риски и открытые вопросы

| Риск | Влияние | Митигация |
|------|---------|-----------|
| Разные LMS по-разному интерпретируют SCORM API (особенно старые SCORM 1.2 реализации) | Трекинг может не сохраняться | Тестовая матрица минимум по 2–3 LMS, fallback-логика в API-обёртке |
| Хотспоты «плывут» при сильно разных разрешениях экрана обучаемого | Ложные ошибки, фрустрация | Жёсткое масштабирование контейнера (`object-fit: contain`), координаты только в относительных единицах |
| Большой вес пакета (много скриншотов высокого разрешения) | Отказ загрузки в LMS с лимитом | Автоматическая компрессия при экспорте, предупреждение методисту |
| Методист случайно создаёт логически противоречивый сценарий (тупиковые переходы) | Обучаемый застревает | Валидатор графа сценария в Author Tool перед экспортом (проверка недостижимых экранов) |

**Открытые вопросы к архитектору:**
- Нужен ли бэкенд на MVP, или можно обойтись клиентским решением с экспортом/импортом проекта как файла?
- Как обрабатывать `learner_id`/имя обучаемого — берём ли из SCORM-данных LMS (`cmi.core.student_name`) для персонализации отчётов, или это не требуется на MVP?

## Приложение A — JSON-схема сценария (пример)

```json
{
  "trainer_id": "legal-address-change-v1",
  "title": "Смена юридического адреса клиента",
  "language": "ru",
  "scorm_version": "1.2",
  "passing_score": 80,
  "screens": [
    {
      "screen_id": "screen_search",
      "image": "assets/screen_1.png",
      "hotspots": [
        {
          "hotspot_id": "input_code",
          "type": "input",
          "region": { "x": 0.32, "y": 0.28, "width": 0.4, "height": 0.06 },
          "expected_value": { "match": "regex", "pattern": "^KZ-\\d{7}$" },
          "on_success": { "next_screen": "screen_client_card", "score": 10 },
          "on_error": { "hint": "Введите код в формате KZ-XXXXXXX", "retry": true }
        }
      ]
    }
  ]
}
```

## Приложение B — минимальный `imsmanifest.xml` (SCORM 1.2, скелет)

```xml
<?xml version="1.0" encoding="UTF-8"?>
<manifest identifier="legal-address-change-v1" version="1"
  xmlns="http://www.imsproject.org/xsd/imscp_rootv1p1p2"
  xmlns:adlcp="http://www.adlnet.org/xsd/adlcp_rootv1p2">
  <organizations default="ORG-1">
    <organization identifier="ORG-1">
      <title>Смена юридического адреса клиента</title>
      <item identifier="ITEM-1" identifierref="RES-1">
        <title>Тренажёр</title>
      </item>
    </organization>
  </organizations>
  <resources>
    <resource identifier="RES-1" type="webcontent" adlcp:scormtype="sco" href="index.html">
      <file href="index.html"/>
      <file href="scenario.json"/>
    </resource>
  </resources>
</manifest>
```
